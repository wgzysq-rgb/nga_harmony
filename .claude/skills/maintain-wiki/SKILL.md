# Wiki 持续维护

## 概述

**修改即更新。** 任何影响模块行为、配置、接口或数据流的代码变更，必须同步更新 Wiki 文档。本 Skill 是 `write-technical-wiki`（新建 Wiki）的补充，定义变更过程中的 Wiki 维护流程。

## 何时使用

- 修改现有功能的代码时
- 新增功能或模块时
- 进行架构变更时
- 修复关键 bug 时
- 修改配置字段/API 参数/消息协议时

## 核心原则

**修改即更新。** 在修改代码的同时或之前更新 Wiki，不要留到"以后"。

## 触发条件与对应操作

| 触发操作 | 必须更新的 Wiki | 更新内容 |
|----------|----------------|----------|
| 新增模块 | 新建对应 Wiki | 完整编写（遵循 write-technical-wiki） |
| 修改配置字段 | 对应配置章节 | 更新字段名、默认值、说明 |
| 修改消息协议 | WebSocket/消息协议章节 | 更新消息类型表、时序图 |
| 修改 API 调用参数 | 核心流程章节 | 更新代码片段、参数说明 |
| 增加/删除依赖 | 系统架构章节 | 更新架构图、依赖关系 |
| 修复关键 Bug | 已知问题/边缘情况 | 补充修复记录 |
| 修改错误处理逻辑 | 错误处理章节 | 更新处理方式、重试策略 |
| 新增配置项 | 配置指南章节 | 添加参数表格 |
| 架构决策变更 | 架构决策目录 | 新增 ADR 文档 |

## 维护流程

### 1. 变更前 — 标记过期

在修改代码前，在对应 Wiki 文件顶部添加过期标记：

```markdown
> ⚠️ **待更新**：此章节因 [#ISSUE_NUMBER / commit HASH] 中的变更需要更新
```

### 2. 变更中 — 同步更新

按照**修改即更新**原则，修改代码与更新 Wiki 同时进行。

### 3. 变更后 — 更新交叉引用

更新可能受影响的关联文档的「关联文档」章节链接。

### 4. 架构决策 — 写 ADR

如果变更涉及架构层面的选择（多个方案对比后选一），必须在 `.wiki/架构决策/` 下新增 ADR 文档。

**ADR 模板：**

```markdown
# 标题（如：为什么选择方案A而非方案B）

## 背景
描述当时面临的问题和约束条件。

## 方案对比
| 方案 | 优势 | 劣势 |
|------|------|------|
| 方案A ✅ | ... | ... |
| 方案B | ... | ... |

## 决策
选择的方案和简要理由。

## 理由
1. 理由一
2. 理由二
3. 理由三

## 代价
选择此方案的权衡之处。

## 参见
- 相关 Wiki 链接
```

## 质量标准

### 每次维护必须检查

- [ ] 配置参数表是否同步（字段名、默认值、说明）
- [ ] 代码引用行号是否仍然有效（`文件:行号`）
- [ ] 架构图是否仍然反映实际数据流
- [ ] 错误处理描述是否匹配当前逻辑
- [ ] FAQ 是否需要更新或增删
- [ ] 关联文档的链接是否仍然有效
- [ ] Mermaid 图语法与复杂度校验通过（`npx mermaid-sonar ".wiki/**/*.md"`，零 error、零 warning）

### Mermaid 图校验

所有 Mermaid 图必须通过 `mermaid-sonar` 校验：

```bash
npm install -g mermaid-sonar         # 首次安装
npx mermaid-sonar ".wiki/**/*.md"    # 校验
```

**校验规则：**

| 规则 | 默认阈值 | 说明 |
|------|----------|------|
| `max-nodes` | 50/100 | 高密度图 ≤ 50 节点，低密度 ≤ 100 |
| `max-edges` | 100 | 超过 O(n²) 复杂度警告 |
| `graph-density` | 0.3 | 图密度超限建议拆分 |
| `max-branch-width` | 8 | 单节点并行分支 ≤ 8 |
| `long-labels` | 40 字符 | 节点标签长度限制 |
| `horizontal-width-readability` | 1200px | 图宽超出视口时改用 LR 或拆分 |
| `layout-hint` | — | 时序流图建议 LR 布局 |
| `disconnected-components` | — | 断开子图需拆分为独立图 |

### 定期维护（每季度）

- [ ] 检查 `.wiki/` 中所有文件的代码引用行号是否仍然有效
- [ ] 删除已废弃功能的 Wiki 或标注为「已废弃」
- [ ] 检查架构决策 ADR 是否有被后续决策推翻的，标注 deprecation

## 自查清单

Wiki 维护完成后逐项确认：

- [ ] 过期标记已移除
- [ ] 配置参数表与实际代码一致
- [ ] 代码引用行号已验证
- [ ] 关联文档链接正确
- [ ] 欢迎阅读.md 的依赖关系表同步更新（如果涉及模块关系变更）
- [ ] 架构决策已归档（如果是架构变更）
- [ ] 如果是新模块，已在 `欢迎阅读.md` 中添加阅读引导
- [ ] Mermaid 图校验通过（`npx mermaid-sonar ".wiki/**/*.md"`，零 error、零 warning）

## 常见错误

| 借口 | 现实 |
|------|------|
| "改动很小，不需要更新 Wiki" | 任何影响行为/配置/接口的变更都需要同步 Wiki |
| "先写代码，回头再补 Wiki" | 回头补 = 永远不会补。代码与 Wiki 同步更新 |
| "用户没说要更新 Wiki" | 修改即更新，无需用户提醒。这是 Skill 职责 |
| "以后重构时一起更新" | 重构遥遥无期。当前变更必须当前更新 |
| "Wiki 改起来太麻烦" | 不改的 Wiki = 废弃的 Wiki，比没有 Wiki 更误导人 |
