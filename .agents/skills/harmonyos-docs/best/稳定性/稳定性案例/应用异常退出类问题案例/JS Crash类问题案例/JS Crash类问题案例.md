 ```
// Update the attributes related to manual effects

  sceneContainerSessionList: SCBSceneContainerSession[]): void {

```

  #### XXX is not callable类案例

 #### [h2]概述

表示应用代码试图像调用函数一样去调用某个对象，但该对象不可被调用（不是一个函数或方法），xxx表示不可被调用对象的值。
- 误将变量名用作函数名
- 对象方法不存在或未正确引用

  该问题主要场景：
- 类被混淆了，开发者调用对应的方法报出的问题。（混淆导致的方法不存在）
- 开发者调用的对象出现了非预期的值或者该对象无相关方法，例如number没有trim方法。
- 开发者自定义的napi，so未加载，调用对应方法不存在而报错。

  #### [h2]分析步骤

通过以上报错堆栈和报错信息，能够定位到具体代码示例如下：```

    a.trim(); // 报错undefined is not callable;

  原因分析：入参是number类型，number类型无trim函数。

  #### [h2]修复方法

这个问题在于b(a)调用时传递的是数字123，但b函数中却试图调用字符串方法trim()，数字没有trim方法，所以传参前将其转换成字符串即可。```
function b(a) {
    a.trim();
}

export function c1() {
    let a = 123;
    b(a.toString()); // 转换为字符串 
}
```

- 检查错误位置：查看错误堆栈确定哪一行代码触发了错误。(如果开启混淆功能下的代码压缩功能，需先还原报错代码栈，具体操作可参考：[报错栈还原](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V14/source-obfuscation-V14#报错栈还原))。
- 检查对象类型：使用typeof操作符来检测数据类型。
- 设置参数默认值：例如为参数设置空字符串默认值，确保始终有字符串可供操作。

  #### taskpool：failed to serialize result类问题案例

 #### [h2]概述

 taskpool执行任务失败，返回了序列化失败的异常信息。这是因为通信对象是不可序列化的类型。

 #### [h2]问题现象

 由于taskpool执行任务失败不会产生JS Crash，所以只能观察hilog日志。而在taskpool序列化结果失败场景下，其会在hilog日志中抛出异常failed to serialize result。

 ```
//在hilog日志中查找
resultError task catch message:An exception occurred during serialization, taskpool: failed to serialize result.
Serialize error: Serialize don't support object type: Function
```
 在catch到的信息里，前一段信息里有“during serialization”，表明是在序列化中。后一段是“failed to serialize result”，表明是返回结果的序列化失败，需要看下返回的结果是什么类型。

 #### [h2]分析步骤

```
@Concurrent
export function printArgs(args: number): number {
  return args
}
```
 ```
import taskpool from '@ohos.taskpool'
import { printArgs } from './utils'
import { BusinessError } from '@ohos.base'

@Concurrent
function createTask(a: number, b: number) {
  let sum = a + b;
  let task: taskpool.Task = new taskpool.Task(printArgs, sum);
  return task; //返回值task任务是无法被序列化，导致问题产生
}

function resultError() {
  let task: taskpool.Task = new taskpool.Task(createTask, 1, 2);
  taskpool.execute(task).then((res) => {
  }).catch((e: BusinessError) => {
    console.error("resultError task catch message:" + e.message)
  })
}

@Entry
@Component
struct Index {
  @State message: string = 'Hello World';

  build() {
    RelativeContainer() {
      Text(this.message)
        .id('HelloWorld')
        .fontSize($r('app.float.page_text_font_size'))
        .fontWeight(FontWeight.Bold)
        .alignRules({
          center: { anchor: '__container__', align: VerticalAlign.Center },
          middle: { anchor: '__container__', align: HorizontalAlign.Center }
        })
        .onClick(() => {
          this.message = 'Welcome';
        })
      Button("resultError")
        .onClick(() => {
          resultError()
        })
    }
    .height('100%')
    .width('100%')
  }
}
```
  上面的例子是在执行的任务中创建了一个task任务，并且返回。但是这个createTask函数返回值task对象是无法被序列化的，从而导致了任务的执行失败。

 目前taskpool只支持的类型参见[序列化支持类型](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/serializable-overview)。

 #### [h2]修复方法

```
@Concurrent
export function printArgs(args: number): number {
  return args
}
```
 ```
import taskpool from '@ohos.taskpool'
import { printArgs } from './utils'
import { BusinessError } from '@ohos.base'

@Concurrent
function createTask(a: number, b: number) {
  let sum = a + b;
  return sum;
}

function resultError() {
  let task: taskpool.Task = new taskpool.Task(createTask, 1, 2);
  taskpool.execute(task).then((res) => {
    let task2: taskpool.Task = new taskpool.Task(printArgs, res);
    console.info("resultError task success")
  }).catch((e: BusinessError) => {
    console.error("resultError task catch message:" + e.message)
  })
}

@Entry
@Component
struct Index {
  @State message: string = 'Hello World';

  build() {
    RelativeContainer() {
      Text(this.message)
        .id('HelloWorld')
        .fontSize($r('app.float.page_text_font_size'))
        .fontWeight(FontWeight.Bold)
        .alignRules({
          center: { anchor: '__container__', align: VerticalAlign.Center },
          middle: { anchor: '__container__', align: HorizontalAlign.Center }
        })
        .onClick(() => {
          this.message = 'Welcome';
        })
      Button("resultError")
        .onClick(() => {
          resultError()
        })
    }
    .height('100%')
    .width('100%')
  }
}
```
 如代码所示，让新的task构造在then中执行，createTask函数里返回的结果设置为能够正常序列化的类型。

  #### [h2]建议与总结

 由于TaskPool在进行数据传送过程中，会对传递的参数和返回的结果进行序列化与反序列化的操作，因此对于这些数据类型的设置需严格，以避免异常问题的发生。

 #### XXX is not initialized类问题案例

 #### [h2]概述

 变量未被初始化发生的原因通常是模块间循环依赖导致的。

 #### [h2]问题现象

异常JS Crash日志示例：

 ```
Device info:xxx
Build info:xxx
Fingerprint:xxx
Timestamp:xxx
Module name:xxx
Version:1.1.10.210
VersionCode:101010210
PreInstalled:Yes
Foreground:No
Pid:16128
Uid:20020026
Reason:ReferenceError
Error name:ReferenceError
Error message:z822 is not initialized
Stacktrace:
Cannot get SourceMap info, dump raw stack:
    at static_initializer (entry|entry|1.0.0|src/main/ets/engine/en/KbdEnLayoutUtils.ts:1:1)
    at func_main_0 (entry|entry|1.0.0|src/main/ets/engine/en/KbdEnLayoutUtils.ts:0:3016)
HiLog:
...
```
  #### [h2]分析步骤

报错xxx is not initialized不一定是循环依赖问题，需要通过hdc命令打开循环依赖检测开关进一步确认，无需重启设备。开关命令如下：

 ```
开启：hdc shell param set persist.ark.properties 0x2000105c
关闭：hdc shell param set persist.ark.properties 0x0000105c
```
 开关打开后复现报错就是第一现场报错，解决即可。如果报错仍是xxx is not initialized，就是文件存在循环依赖。

 循环依赖原理：

 
- 模块加载顺序：根据ECMA规范，会按照深度优先的顺序进行加载。假设应用存在加载链路A->B->C，那么它的执行顺序则为C->B->A。
- 循环依赖：如果应用存在加载链路A->B->A，根据深度遍历执行顺序，执行流程会先标记A的状态为加载中，然后去加载B，标记B的状态为加载中，然后去加载A，由于A文件已经标记加载中，根据规范定义，识别到加载中模块会直接返回，就会先执行B文件。

 部分循环依赖会产生crash的原因：

 由上面的叙述可知，B文件依赖A文件，但是B文件先执行，如果B文件导入的A文件变量没有在全局或者静态类中被使用，B文件就会正常执行。如果B文件变量在全局或者静态类中被使用，导致文件执行时就会用到A的变量，就会产生xxx is not initialized的crash，即循环依赖导致变量未被初始化。

  反例：

 ```
import { Animal } from './XXXIsNotInitializedCase1'

export let a = "this is A";

export function A() {
  return new Animal;
}
```
 ```
import { a } from './XXXIsNotInitializedCase'

export class Animal {
  static {
    console.log("this is in class");
    let str = a; // 报错信息：a is not initialized
  }
}
```
 正例:

 ```
import { Animal } from './XXXIsNotInitializedFix1'

export let a = "this is A";

export function A() {
  return new Animal;
}
```
 ```
import { a } from './XXXIsNotInitializedFix'

export class Animal {
  static {
    console.log("this is in class");
  }
  str = a; // 修改点
}
```
 #### [h2]修复方法

循环依赖的检测方法：安全规则[@security/no-cycle](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/ide_no-cycle)。

  #### [h2]建议与总结

循环依赖的根本原因是模块/类之间的职责边界不清晰、耦合度过高。因此，最佳的解决方式是预防优于治疗，通过良好的设计来避免其产生。

  #### Error类案例

 #### [h2]概述

Error类问题一般是开发者或JS库主动抛出来的JS异常。

 这类问题目前有两种场景：

 
- 应用程序遇到无法解决的故障，需要考虑抛出JS异常来终止业务并生成故障日志。
- 依赖使用JS库或者其他模块的接口，若可能抛出异常，需要考虑使用try-catch机制进行捕获，否则会终止当前业务。

  #### [h2]问题现象一

 开发者自主抛出JS异常，可通过如下代码实现：

 ```
// In a production environment, encrypted error codes should be used instead of plain text descriptions.
throw new Error("TEST JS ERROR")
```
 通过DevEco Studio的FaultLog工具可收集到此类异常日志，其中JS异常栈信息可直接定位到抛异常的代码位置。

 

 这类问题可通过故障日志定位到具体的代码行，检视上下文来分析问题即可。

 #### [h2]问题现象二

因未处理三方接口抛出的JS异常导致JS Crash日志示例：```
Error name:Error
Error message:BusinessError  2501000: Operation failed.
Error code:2501000
Stacktrace:
Cannot get SourceMap info, dump raw stack:
  at onStart (product/phone/build/default/cache/default/default@CompileArkTS/esmodule/release/feature/systemstatus/linkspeedcomponent/src/main/ets/default/controller/NetSpeedController.ts:50:1)
  at NetSpeedController (product/phone/build/default/cache/default/default@CompileArkTS/esmodule/release/feature/systemstatus/linkspeedcomponent/src/main/ets/default/controller/NetSpeedController.ts:43:43)
  at getInstance (product/phone/build/default/cache/default/default@CompileArkTS/esmodule/release/staticcommon/basiccommon/src/main/ets/component/utils/SingletonHelper.ts:17:17)

```
   #### [h2]分析步骤

 
提取日志关键信息通过日志信息可以确定为Error类问题，为代码主动抛出的异常。可以通过异常代码调用栈，获取错误产生位置。Cannot get SourceMap info, dump raw stack信息表示该应用为release包安装，JS栈转换eTS行列号失败，可考虑使用[堆栈轨迹分析](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/ide-release-app-stack-analysis)来解析行号。

 定位到具体代码通过以上 JS 堆栈，能够定位到NetSpeedController.ts文件中具体代码片段如下，异常抛出位置为wifiManager.on函数调用。

 ```
wifiManager.on('wifiConnectionChange', (data) => {
  this.isConnected = data === 1 ? true : false;
  this.handleUpdateState();
});
```
 
 #### [h2]修复方法

通过分析wifiManager.on源码，得知该函数内存在部分场景会抛出内容为BusinessError 2501000: Operation failed.的JS异常，识别此类异常不会导致程序无法运行下去，考虑使用try-catch机制对异常进行捕获处理。具体的修改方法可参考如下:

 ```
try {
  wifiManager.on('wifiConnectionChange', (data) => {
    this.isConnected = data === 1 ? true : false;
    this.handleUpdateState();
  });
} catch (error) {
  log.showError('wifiConnectionChange error');
}
```
  #### [h2]建议与总结

对于这类问题，在编码阶段使用可能会抛异常的接口时，考虑是否需要捕获该异常，避免影响应用主体业务。

  #### OutOfMemoryError类案例

 #### [h2]概述

OutOfMemoryError类问题一般是由于程序在运行过程中，因申请的ArkTS内存超过运行时环境可分配的内存限制，导致内存分配失败而触发的错误。

  #### [h2]问题现象

Error name:OutOfMemoryError字段代表应用存在内存泄漏的场景，对应的崩溃堆栈如下：

 ```
Device info:xxx
Build info:xxx
Fingerprint:873b2d78dbf1c413e983bae646887824217f21ad45746f8cd70beafc8212c482
Timestamp:xxx
Module name:com.example.oom
Version:1.0.0
VersionCode:1000000
PreInstalled:No
Foreground:Yes
Pid:20168
Uid:20020198
Process Memory(kB): 663416(Rss)
Device Memory(kB): Total 11679260, Free 898896, Available 5358592
Reason:OutOfMemoryError
Error name:OutOfMemoryError
Error message:OutOfMemory when trying to allocate 40 bytes function name: SharedHeap::AllocateOldOrHugeObject 
Stacktrace:
Cannot get SourceMap info, dump raw stack:
    at testGenStringOOM entry (entry/src/main/ets/pages/Index.ts:39:15)
    at anonymous entry (entry/src/main/ets/pages/Index.ts:21:13)
```
  #### [h2]分析过程

 常规的JsCrash日志通过其清晰的调用堆栈，能够精准地定位到引发异常的根源代码。相比之下，OutOfMemory异常则不同，其相关的JsCrash日志往往是内存问题长期积累后的最终表现，而非一个直接的触发点。因此，OutOfMemoryError类异常JsCrash日志通常无法提供立竿见影的诊断价值，需要借助Snapshot内存占用分析工具来分析根本原因。

 从上文JsCrash日志中可以看到：应用名、pid、报错信息、调用栈等，其中有参考意义的主要是报错信息，这个报错信息告诉我们，这个Crash是因为OldSpace+HugeSpace的和到达了上限，这时候尝试再往Space中分内存就发生了OOM。调用栈的参考意义不大，调用栈只说明当前分配了内存，如果到达上限，分1个字节的内存也会OOM，所以一般情况下调用栈可以忽略。

 特殊情况：有些应用高频的发生OOM，而且每次的Crash文件的调用栈都一样，那说明这个调用栈可能是个高频调用，而且有泄露，最终导致了OOM，这种情况才有查看调用链的价值，其它情况只需要分析Heapsnapshot文件即可。

 下面介绍两种形态如何分析：

- ArkTs内存管理方式和GC的时机

 首先要明确方舟虚拟机是如何管理ArkTs内存，并且在什么时候进行GC的。

 在方舟虚拟机采用的是标记清除法回收内存，内存是通过维护一棵树来管理的，这颗树的根节点被称为GC Root，ArkTs对象只要被挂载到这颗树上则说明该对象存活。很多刚接触内存的同学可能会对GC Root产生误解，认为挂在GC Root上的对象是需要被回收的，而根节点之所以叫GC Root是因为标记清楚法在标记阶段，垃圾收集器从GC Root开始遍历。所有从GC Root可达的对象都会被标记为“存活”。在清除阶段，垃圾收集器会扫描内存中的所有对象。如果某个对象没有在标记阶段被标记为存活，那么该对象就是“垃圾”，可以被回收。垃圾收集器会释放这些未标记对象所占用的内存空间。每次GC都是从根节点开始，所以根节点被称为GC Root。该树的快照如图所示：

 

 ArkTs的对象是存在引用和被引用关系的，图中箭头指向的都是被引用对象，所有存活的对象都直接或间接被GC Root所引用，从GC Root到被引用对象的路径为该对象的引用链。因此，解决内存泄漏的关键就是在于将对象的引用链在合适的地方（结合业务判断）断开。

  #### Can not get Prototype on non ECMA Object类案例

 #### [h2]概述

在ArkTS与Native交互开发中，存在不当的生命周期管理操作。具体表现为，将仅在当前调用栈有效的napi_value句柄直接存储于Native侧的全局变量中。当此句柄伴随返回值传递至ArkTS侧继续使用时，由于其原始依赖的上下文已超出生命周期且未通过napi_ref建立持久化引用，将导致句柄失效，引发未定义行为。

 该问题主要场景：

 
- 全局变量存储裸napi_value
- 跨异步回调传递裸napi_value
- 将napi_value作为回调参数持久化
- 在多线程环境中共享napi_value

  #### [h2]分析步骤

通过以上报错堆栈和报错信息，能够定位到具体代码示例如下：

 ```
import { hilog } from '@kit.PerformanceAnalysisKit';
import testNapi from 'libentry.so';

declare class ArkTools {
  static hintGC(): void;
}

@Entry
@Component
struct Index {
  @State message: string = 'Hello World';

  build() {
    Row() {
      Column() {
        Text(this.message)
          .fontSize(50)
          .fontWeight(FontWeight.Bold)
          .onClick(() => {
            hilog.info(0x0000, 'testTag', 'Test NAPI 2 + 3 = %{public}d', testNapi.add(2, 3));
            ArkTools.hintGC(); // 触发GC
            // setTimeout(()=> {  // 异步方法
            testNapi.getValue().length;
            // }, 1000);
          })
      }
      .width('100%')
    }
    .height('100%')
  }
}
```
 ```
napi_value objValues;
static napi_value Add(napi_env env, napi_callback_info info) {
    size_t argc = 2;
    napi_value args[2] = {nullptr};

    napi_get_cb_info(env, info, &argc, args, nullptr, nullptr);

    napi_valuetype valuetype0;
    napi_typeof(env, args[0], &valuetype0);

    napi_valuetype valuetype1;
    napi_typeof(env, args[1], &valuetype1);

    double value0;
    napi_get_value_double(env, args[0], &value0);

    double value1;
    napi_get_value_double(env, args[1], &value1);

    napi_value sum;
    napi_create_double(env, value0 + value1, &sum);

    napi_status status;
    napi_handle_scope scope;
    status = napi_open_handle_scope(env, &scope);
    napi_value jsArray = nullptr;
    napi_create_array(env, &jsArray);
    // 将创建好的数组进行赋值
    for (int i = 0; i < 5; i++) {
        napi_value element;
        napi_create_int32(env, i, &element);
        napi_set_element(env, jsArray, i, element);
    }
    status = napi_close_handle_scope(env, scope); // 作用域关闭，jsArray 被清理
    objValues = jsArray;                          // 问题所在！

    return sum;
}

static napi_value GetValue(napi_env env, napi_callback_info info) { return objValues; }
```
 在当前实现中存在一个严重的变量生命周期管理问题。核心问题在于全局变量objValues存储的是一个napi_value引用，该引用在Add函数中被赋值后立即失效。

 具体来说，当第一次调用Add函数时：创建jsArray数组对象并将其赋值给全局变量objvalues，然后关闭局部作用域napi_handle_scope，这使得jsArray的引用计数减少，成为可被垃圾回收的状态。当后续调用ArkTools.hintGC()强制触发垃圾回收时，jsArray的实际内存被释放，但全局变量objValues仍然持有这个已被释放的引用地址。此后，无论是调用GetValue()函数还是再次访问objValues，都会返回一个悬空指针，这种访问已释放内存的行为导致未定义结果，通常引发程序崩溃或不可预知的内存错误。

  #### [h2]修复方法

避免在NAPI中使用全局变量来存储napi_value，改为napi_ref来管理跨作用域的对象引用

 ```
static napi_ref objvalues = nullptr; // 使用napi_ref来管理跨作用域的对象引用
```
  #### [h2]建议与总结

在ArkTS与Native交互开发中，napi_value对象引用的生命周期管理至关重要。通过napi_value模块接收的Native指针本质上是对原生对象的临时访问句柄，仅在当前调用栈和ArkTS对象存活期内有效。如需跨异步操作或对象生命周期持久化引用，必须使用napi_create_reference创建强引用，以确保垃圾回收机制识别该对象仍在被ArkTS代码持有。

 若直接将裸napi_value存储于全局变量或长生命周期对象中，短期内可能看似正常，但一旦底层原生对象被释放或ArkTS运行时执行内存回收，该引用即变为悬挂指针，导致访问异常、内存错误或数据混乱。此类问题具有隐蔽性，可能在特定触发条件下才显现，但必然在相关资源释放后导致不可预测行为。因此，开发中必须遵循：仅在同步回调内临时使用Native指针；需长期持有则必须通过napi_ref机制实现；使用完毕后需及时释放引用；同时结合异常处理机制确保资源安全。
