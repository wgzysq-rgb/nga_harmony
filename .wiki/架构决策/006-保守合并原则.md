# 为什么重构中"行为保持"优先于"字面合并"（保守保留重复代码）

## 背景

17 项模块可维护性重构计划多处出现"合并 / 统一"目标：合并 4 份 `decodeHtmlEntities`、统一 `resolveAttachUrl`、`loginPassword` 走统一传输通道、合并 `@Builder` 渲染片段等。但在实施过程中发现，强行合并若干"看似重复"的代码会改变运行时行为，与"行为保持"的重构约束相冲突。典型案例包括：

- **`decodeHtmlEntities`（两版语义不同）**：BBCode 正文版除反转 HTML 实体外，还需把 `<br>` 转为换行、剥离残余 HTML 标签（`parser/_shared/HtmlEntityCodec.ets:70-71`），因为 BBCode 解析器以纯文本切片为单位处理；而纯文本字段版（用户名、标题、签名等，`HtmlEntityCodec.ets:32-47`）仅反转实体，不触碰 `<` 与标签，套用前者会吞掉合法的 `<`。
- **`resolveAttachUrl`（两版 CDN host 与纯数字判断不同）**：`ContentParser` 路径需对多 CDN host 兜底，`BBCodeParser` 路径则原样保留，两版正则与 host 列表不同。
- **`loginPassword` 需读 response header**：登录成功后需从 `response.header['set-cookie']` 提取会话 cookie（`service/NgaClient.ets:519-521`），而统一通道 `httpReq` / `executeWithRetry` / `ngaRequest` 的返回类型是 `{ status, body: ArrayBuffer }`（`NgaClient.ets:158-161`），不透传 response.header。强行让 `loginPassword` 走统一通道会丢失 set-cookie，登录态失效。
- **`BBCodeContentView` 的 `@Builder` 与 `@State` 深度耦合**：约 18 个 `@Builder` 与所在 struct 的 `@State` 紧绑，ArkUI 不支持跨 struct 调用 `@Builder`（需拆组件改用 `@BuilderParam`），强行外迁会改变渲染闭包与状态观测关系。

这类情况在重构中反复出现，需要确立一条统一处理原则。

## 方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| 行为保持优先（保守保留 + 记 follow-up + commit 说明理由） ✅ | 零运行时行为变更、可后续渐进合并、风险可控、与"无单测 + 设备手动回归"现状匹配 | 保留视觉上的重复（短期债务） |
| 字面合并（强行统一所有看似重复） | 代码字符数减少、视觉整洁 | 改变运行时行为、需大量设备回归、无单测网兜底、易遗漏 corner case |

## 决策

确立"行为保持优先于字面合并"原则：当合并 / 统一会改变运行时行为时，保守保留重复代码 + 记录为 follow-up + 在 commit message 与重构报告中说明保留理由。只有在不改变行为的前提下，才执行字面合并。

## 理由

1. 本次重构的目标是可维护性（降低重复、理清分层、归一命名），不是单纯减少字符数；运行时行为变更是另一个维度的风险，不应混入"行为保持"的重构 commit，否则会把可维护性改动与潜在 bug 混在一起，难以二分排查。
2. 项目无单测基建，验证依赖 DevEco 编译通过 + 设备手动回归，行为变更的回归成本高、易遗漏 corner case（如 `<br>` 被吞、set-cookie 丢失、CDN host 不匹配）。
3. 保守保留可后续在充分测试与传输层扩展的前提下渐进合并。例如 `loginPassword` 需先扩展 `httpReq` / `executeWithRetry` 透传 `response.header`（返回类型从 `{ status, body }` 扩展为 `{ status, body, header }`），再让登录走统一通道——这是一个独立的、有明确验证点的后续改动，不应在重构 commit 中顺带完成。
4. 典型案例可复用决策：`decodeHtmlEntities`（BBCode 版 vs 纯文本版语义不同）、`resolveAttachUrl`（ContentParser 多 host 兜底 vs BBCodeParser 原样）、`loginPassword`（需 response.header）、`@Builder`（ArkUI 跨 struct 调用限制 + `@State` 闭包耦合）。遇到新的"看似重复"代码时，先验证两版语义是否等价，等价才合并，不等价则按本原则保留。
5. 该原则与 ArkTS"类布局编译时固定"约束呼应：迁移类不改字段类型 / 顺序 / 名称，同样是"行为保持优先"的体现——重构中只搬不改，把"改"留给独立、可独立验证的后续 commit。

## 代价

- 保留视觉上的重复代码（短期债务，需 follow-up 清单跟踪真正可合并项）。
- 新读者可能误以为重复是重构遗漏，需 commit message 与本 ADR 解释保留理由，存在认知成本。
- follow-up 清单需持续维护，避免保守保留无限累积成永久债务；定期评估每个保留项是否已具备合并条件（如传输层已透传 header、已补单测）。
