# 备份恢复

## 概述

`entrybackupability/EntryBackupAbility.ets` 是应用的备份扩展能力入口，继承自 `BackupExtensionAbility`（`@kit.CoreFileKit`），由 HarmonyOS 系统在备份/恢复场景下自动触发。

> 当前为骨架实现，`onBackup` 和 `onRestore` 两个回调仅输出日志，尚未填充实际的备份/恢复业务逻辑。

## 生命周期回调

### onBackup

`EntryBackupAbility.ets` — 系统触发备份时调用：

```typescript
// 备份回调 — 当前仅日志记录
async onBackup(): Promise<void> {
  logger.info('onBackup ok')
}
```

当前实现无实际备份逻辑。后续应在此处将 AppStore 的持久化数据（Preferences）、缓存文件等打包为备份归档。

### onRestore

`EntryBackupAbility.ets` — 系统触发恢复时调用，接收 `BundleVersion` 参数标识备份版本：

```typescript
// 恢复回调 — 当前仅日志记录
async onRestore(bundleVersion: BundleVersion): Promise<void> {
  logger.info('onRestore ok %{public}s', JSON.stringify(bundleVersion))
}
```

## 备份策略

### 当前状态

- **数据持久化**：`AppStore.flushAll()` 在 `onBackground` 时触发（详见 [应用生命周期](应用生命周期.md)）
- **备份触发**：由 HarmonyOS 系统调度，应用层不主动调用
- **前后台联动**：`onBackground` 时已触发 `flushAll()`，可在备份时确保写入持久化的数据是完整的

### 待实现

- `onBackup`：收集 Preferences 文件 + 缓存数据 → 写入备份归档
- `onRestore`：从备份归档恢复 Preferences 文件 + 重建缓存
- 增量备份策略：避免全量备份造成性能开销

## 错误处理

### 备份/恢复失败

当前骨架实现不涉及实际文件操作，因此无错误处理。后续实现时需注意：

1. **文件读写权限**：备份目录可能需要 `ohos.permission.BACKUP` 权限
2. **数据版本兼容**：`BundleVersion` 标识备份版本，恢复时需检测版本差异，防止旧版本备份覆盖新版本数据
3. **恢复失败降级**：恢复 `onRestore` 失败时不应影响应用正常启动，应回退到本地已有的持久化数据

## 常见问题

**Q: 什么时候会触发 onBackup？**
A: HarmonyOS 系统在应用卸载前或系统设置中触发备份时调用。开发者无法手动触发，由系统调度。

**Q: onRestore 的 bundleVersion 是什么格式？**
A: `BundleVersion` 包含应用的版本信息（versionName、versionCode 等），用于判断备份数据的兼容性。

**Q: 当前备份恢复功能生效吗？**
A: 当前为骨架实现，`onBackup` 和 `onRestore` 仅输出日志，无实际数据备份/恢复逻辑。应用的持久化数据（Preferences）由 `AppStore.flushAll()` 在 `onBackground` 时保障。
